# LEARNING — Harness × Plexer architecture: the binding direction

**Status: BINDING. This direction is settled. Any change that deviates from it does not land.**
This entry exists because the direction was researched, agreed, *claimed built, and then silently abandoned at the call sites* — twice counting the lost original research report. It will not happen a third time.

## The direction, in one paragraph

orch is an **a × b system**: agent harnesses (pi, claude, codex, …) × multiplexers (herdr, tmux, headless, …). Every combination a user composes must work, be reconfigurable per-axis at any time, and be verifiable. This is achieved by a **layered pattern stack** — not any single pattern: Hexagonal ports owned by core (L0), Bridge between the two independently-varying hierarchies (L1), one GoF Adapter per foreign tool owning that tool's entire wire surface (L2), Strategy-with-capability-negotiation inside each axis (L3), a Provider Model registry/factory/builder that composes the configured pair (L4), a single Facade dispatcher as the sole invoker of adapter strategies (L5), and static checks that make every one of those boundaries mechanically un-crossable (L6). Full reference with per-level rules and conformance table: `docs/reference/design-patterns.md`. Gap audit and ordered fix list: `docs/reviews/architecture-review-2026-07-16.md`.

## The laws (violating any of these = the failure mode this entry forbids)

1. **No pair code.** Nothing — no class, function, file, or branch — may represent an (agent, plexer) pair. New harness or plexer = one adapter file + registration + its `installShim()`. Zero call-site edits.
2. **Wire formats live in exactly one adapter.** pi's `inbox.jsonl`/`answer.json`, claude's hooks, codex's notify events, herdr's socket verbs, tmux's argv grammar: each appears in one module. Core reads presence files and calls ports; it never touches a provider's private format.
3. **Callers branch on capabilities, never identity.** `if (caps.steer === "keys")` yes; `if (adapter.id !== "pi")` never. An unsupported capability fails loudly (exit 1), never silently no-ops. This is LSP/terraform-style capability negotiation and it is how genuinely-different integration surfaces get absorbed without leaking.
4. **One dispatcher.** All control-plane traffic (steer/answer/model/dispatch) flows through the single control dispatcher: resolve adapter → gate on caps → invoke port method → execute the returned `AdapterCommand` → backend fallback per caps. The daemon and every command call it; nothing else invokes adapter strategies.
5. **Composition, never baked.** The (adapter, backend) pair is composed at config-resolution time for new spawns and recorded per-agent at spawn time (identity key + spawn registry). Live agents keep their spawn-time pairing; changing defaults affects new spawns only; mixed fleets are a supported steady state. Re-pairing is a settings write — `installShim()` is idempotent and additive, dormant shims self-gate, no uninstall/reinstall ceremony.
   **Setup is multi-select (2026-07-16, Bryan):** the wizard installs a SET of providers per axis (e.g. pi AND claude; multiple plexers), not one. `settings.json` records the installed set per axis plus the currently-active `defaults.adapter`/`defaults.backend`, which must be members of the installed set; switching the active provider among installed ones at any time is a plain settings edit (wizard or hand). `--agent`/`--backend` per-spawn overrides may use any installed provider. Doctor verifies every INSTALLED provider's integration, not just the active pair.
6. **User-facing `$ORCH_DIR/settings.json` is the composition's storage.** Written by setup via whole-file JSON round-trip, hand-editable, `schemaVersion`-stamped, validated loudly on every load. Exactly one current schema (repo Rule 8) — no migrations, no legacy aliases. Replaces `config.toml`, the hand-rolled TOML parser, the `Bun.TOML` probe, and `writeDefaultEntry` text surgery.
7. **Verification is first-class.** `orch doctor` checks declared settings against reality: configured backend available/inside-session, configured adapter on PATH with its shim installed and current, every pair still live in the fleet. `orch settings` prints effective values with provenance (flag > env > settings.json > default). Checks derive from the resolved providers, never a hardcoded list.
8. **New concerns are new axes, never new combinations.** A 3rd/4th independent concern (notifiers, webhooks, MCP, auth, …) = its own port + provider registry + `settings.json` key, composed only at the composition root, interacting with other axes only via core domain events. Cost stays additive (Σ providers), never multiplicative (Π combinations). A provider module may import core types only — never another provider family. Notify sinks already prove this in-repo (third axis, zero cross-axis knowledge, stayed clean). See `docs/reference/design-patterns.md` L1b.
   **Sequencing precondition: NO new axis lands while an existing axis is out of conformance.** The additive guarantee only holds when the laws hold. In the 2026-07-16 state (pi wire format smeared across core, no dispatcher, dead installShim, id-branches at call sites) a new axis would have been forced to couple to pi/herdr internals just to function — inheriting and multiplying the rot. Close the L2/L3/L5/L6 gaps first, always.
9. **Static enforcement or it didn't happen.** `check:bridge` guards the axis wall; the port-boundary check guards core against concrete adapter/backend imports and wire-format literals; both are CI gates and openspec migration gates. **An openspec change is not "done" until its scenarios have actually been executed** — the skipped verification gate (task 8.2) is precisely how "built" was falsely claimed last time.

## Why this entry exists (the failure record)

- The original GoF/architecture research report was commissioned, produced, and **never committed** — only a five-line summary survived in `openspec/changes/pluggable-plexer-backends/design.md`. Lost work, re-done 2026-07-16.
- The plexer axis of the design was executed with migration gates and a static check — and stayed clean. The agent axis got the same interfaces but **no enforcing decision, no dispatcher, no check** — and every call site inlined pi's happy path: `orchd.ts` imported `piAdapter` directly and hardcoded its inbox for steer/model; adapter methods (`detectState`/`extractResult`/`installShim`) went entirely uncalled; codex shipped as a shell with zero presence wiring; claude steers were silently dropped while reported delivered.
- Lesson, distilled: **interfaces are not architecture. The context that invokes them, the factory that assembles them, and the static check that guards them are the architecture.** A pattern without its enforcement level regresses to inlining under delivery pressure, every time.
