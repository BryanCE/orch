## Why

The `AgentAdapter` port exists but nothing in core invokes it as the control authority: the daemon imports `piAdapter` directly and hardcodes pi's `inbox.jsonl` for every steer and model change, so a claude or codex agent is told a steer "succeeded" while the write lands in a file it never reads. Adapter `steer()`/`answer()`/`extractResult()` are dead code, and the one place that does call `steer()` (`commands.ts:1149`) branches on `adapter.id !== "pi"` and then drops the returned `AdapterCommand` on the floor — codex's real `codex resume` argv is built and discarded. This is the L5 (single dispatcher) and L6 (port-boundary check) gap the architecture review names as the worst defect: the strategies were built and the context that invokes them was skipped, so the pi wire format leaked across core and every non-pi pairing is silently broken. It must close before any new axis lands (learnings law #8 sequencing precondition).

## What Changes

- **Add a single control-plane dispatcher** (the L5 Facade/Context) — the sole invoker of adapter control strategies, **running inside the daemon**. `deliverControl(target, action)` (for the steer and model actions; `orch answer` calls `adapter.answer` directly in the CLI instead) resolves the target's adapter from its presence/registry record, gates on `adapter.caps`, invokes the adapter's steer/model mechanism, **executes the returned `AdapterCommand` argv itself daemon-side via `node:child_process`** (currently dropped) — treating a nonzero exit as a failed steer — falls back to `backend.deliver({kind:"message"})` when `caps.steer === "keys"`, and exits 1 loudly when the capability is absent — never a silent no-op. It is invoked only by the daemon's RPC handlers, after governance; the CLI reaches it over the orchd socket.
- **Route the daemon through the dispatcher.** **BREAKING** (internal): delete `import piAdapter` from `orchd.ts`, the hardcoded `piAdapter.steer` branch in `deliverBackend`, and the direct `inbox.jsonl` append in `setModel`; `setModel` gates on `caps.setModel` and fails fast when unsupported.
- **Route the command layer correctly.** The brokered verbs — `cmdSteer` (including its presence-only branch, which today calls `adapter.steer` in-process at `commands.ts:1149`), `cmdBroadcast`, and the steer leg of `cmdPipe` — `writeRpc` to orchd so the daemon's dispatcher (with caps gating and returned-command execution) applies the effect; the `id !== "pi"` check is deleted daemon-side, not re-implemented in the CLI. The read/local verbs — `cmdAnswer`, `cmdResult`, and `cmdPipe`'s result extraction — call `adapter.answer` / `adapter.extractResult` directly instead of reading and writing agent files.
- **Contain pi's wire format** (L2 / law #2). Move all `inbox.jsonl` and `answer.json` reads and writes into `src/adapters/pi.ts`; those filenames appear in exactly one module. Core reads presence and calls ports only.
- **De-pi the lifecycle verbs** (L2/L3). `orch reset`/`reload`/`restart` (`cmdNew`/`cmdReload`/`cmdRestart`, `commands.ts:2210-2410`) hardcode pi's `/new`/`/reload`/`/quit` slash-commands — meaningless for claude/codex and owned by no other change. Move those strings into `src/adapters/pi.ts` behind a declared lifecycle capability; core resolves the adapter, gates on the capability, and fails loudly (exit 1) when a target's adapter declares no lifecycle mechanism, instead of delivering a foreign keystroke. The protocol-neutral readiness polling stays in core.
- **Add a port-boundary static check** (L6). Extend `scripts/check-bridge.ts` (or a sibling): core outside `src/adapters/` and `src/backends/` may not import a concrete adapter/backend module, may not contain any adapter wire literal from the single exhaustive set the check owns (pi's `inbox.jsonl`/`answer.json`, codex's notify events `agent-turn-complete`, claude's hook identifiers/paths), and may not contain adapter/backend id-equality branches. Wired into `bun run check` and CI.

Non-goals / deferred to other changes (referenced by name, not done here): codex/claude presence writers (`adapter-presence-writers`); setup/doctor provider-driven checks (`provider-driven-setup-doctor`); `settings.json` replacing `config.toml` (`settings-json-config`); tmux backend completion (`tmux-backend-completion`); the `commands.ts`/bridge monolith split (`monolith-file-breakdown` — the dispatcher lands as its own small module so that split adopts it). This change wires the dispatcher and containment against the current file layout; it does not move command bodies into `src/commands/`.

## Capabilities

### New Capabilities
- `control-dispatch`: the single, daemon-side control-plane dispatcher that owns all agent-directed control traffic (steer, model) — resolve adapter, gate on capability, invoke the port method, execute the returned `AdapterCommand` via `node:child_process`, fall back to the backend per capability, and fail loudly when the capability is absent or the command exits nonzero. Establishes the dispatcher as the sole invoker of adapter control strategies; the daemon's RPC handlers are its only callers, and brokered CLI commands reach it over the orchd socket. Also brings the lifecycle verbs (`reset`/`reload`/`restart`) onto a declared adapter lifecycle capability so they stop hardcoding pi's slash-commands.
- `port-boundary-guard`: the static enforcement that keeps core free of concrete adapter/backend imports, adapter wire-format literals (the exhaustive set — pi's `inbox.jsonl`/`answer.json`, codex's `agent-turn-complete`, claude's hook identifiers — kept single-sourced in the check script), and adapter/backend id-equality branches — run as part of `bun run check` and CI so the boundary is mechanically un-crossable.

### Modified Capabilities
- `agent-adapters`: strengthen the presence-protocol and capabilities requirements so that (a) an adapter's private wire format lives only inside that adapter (pi's `inbox.jsonl`/`answer.json`), and (b) a control command that receives an `AdapterCommand` from a port method MUST execute it rather than discard it.
- `dispatch-broker`: the daemon applies steer/model effects by calling the control dispatcher, not by importing a concrete adapter or appending an agent inbox file directly; the requirement text drops the pi-inbox and "herdr send" wording in favor of the adapter-neutral dispatcher path.

## Impact

- Code: `src/daemon/orchd.ts` (remove `piAdapter` import, rewrite `deliverBackend`/`setModel`), a new dispatcher module (e.g. `src/control/dispatch.ts`), `src/adapters/pi.ts` (absorb `inbox.jsonl`/`answer.json` I/O from `src/store.ts`, plus the `/new`/`/reload`/`/quit` lifecycle strings behind a declared capability), `src/store.ts` (remove `steerPresence`/`writeAnswer` inbox/answer writers), `src/commands.ts` (`cmdSteer`, `cmdAnswer`, `cmdResult`, `cmdPipe`, `cmdBroadcast`, and the `reset`/`reload`/`restart` lifecycle verbs), `scripts/check-bridge.ts` (port-boundary check with the broadened wire-literal set), `package.json` scripts + CI.
- Behavior: steer/model/answer/pipe/broadcast become adapter-correct for claude and codex — a returned `AdapterCommand` is executed, an unsupported capability fails with exit 1 and an actionable message. pi behavior is unchanged (still `inbox` steer, still `orch_ask` answers, still model-via-inbox), now through the dispatcher.
- No runtime dependencies added; node-safe only (repo Rule 6). No back-compat shims — old call sites are rewritten and deleted (repo Rule 8).
