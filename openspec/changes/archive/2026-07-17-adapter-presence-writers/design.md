## Context

The `AgentAdapter` port (`src/adapters/adapter.ts`) already declares `detectState`, `extractResult`, and `installShim?`. Only pi honors them end to end via `extensions/orchestrator-bridge.ts`. Claude has a settings.json hook shim (`scripts/claude-hooks.ts`) but its installer (`installClaudeHooks`) sits inline in `commands.ts:1553` and is called by a `harness === "claude"` branch, not through the port. Codex ships tested parsers (`codex.ts:219,250`) that nothing invokes and no presence writer at all. The view layer hardcodes pi: `deriveView` (`commands.ts:110-148`) calls `parseSession` only when `ent.agent === "pi"`, presence-only entities are labeled `agent:"pi"` (`entities.ts:120`), the `Entity.herdrStatus` field name is a lie (it's populated generically), and `WORKER_PROMPT_HEADER` (`commands.ts:2170`) bakes pi's `orch_ask` into every worker prompt.

Constraints that shape every decision below: presence writers are bundled leaf modules that must stay node-safe and must not drag `store.ts`'s sqlite graph into their bundles (`src/presence-schema.ts` header); the codex shim runs under whatever runtime the user has (node/deno/bun), exactly like the claude shim; each foreign tool's wire format lives in exactly one adapter module (law #2); callers gate on `caps`, never on adapter id (law #3); re-running any `installShim()` is additive and idempotent (law #5). The presence-dir file protocol and `PRESENCE_SCHEMA` are frozen — new writers stamp the same schema.

## Goals / Non-Goals

**Goals:**
- A codex presence writer: a notify shim on `agent-turn-complete` for interactive/pane codex, and adapter-parsed session tail for headless codex — both routed through the existing `CodexAdapter` parsers.
- A concrete `installShim()` on every adapter (pi/claude/codex), each installing only its own integration, additively and idempotently.
- The view/presence layer reads adapter-declared capabilities instead of `if (agent === "pi")`: session/state fallbacks come from the resolved adapter (gated on `caps.sessionTail`), the agent label comes from the presence record, and the worker prompt references `orch_ask` only when `caps.ask`.
- Honest claude presence: the coarse hook granularity is documented and specced as start/stop/notification only.

**Non-Goals:**
- The control dispatcher and routing of `orch steer`/`model`/`answer`/`result`/`pipe` through the port (**adapter-control-authority**). This change wires the parsers into the *view/presence* path only, not the result/control commands.
- `orch setup`/`doctor` *calling* `installShim()` and deriving checks from the resolved pair (**provider-driven-setup-doctor**). Here we only *provide* the implementations on the port; the existing `harness === "claude"` call site stays until that change relocates it.
- Splitting `commands.ts` or the bridge into per-domain files (**monolith-file-breakdown**). Edits land in the current files.

## Decisions

### D1 — Codex notify shim mirrors the claude hook shim
A new `scripts/codex-notify.ts`, bundled by a `build:notify` script to `dist/scripts/codex-notify.js` (target=node, format=esm), is the codex analogue of `scripts/claude-hooks.ts`. Codex invokes its `notify` program with a single JSON string argument describing the event; the shim reads `argv[2]`, parses it, and — keyed off `ORCH_AGENT_KEY` (parsed through the one identity boundary, `src/backends/identity.ts`) — writes `status.json` and, on a completed turn with assistant text, `result.json`, using atomic tmp+rename and stamping `PRESENCE_SCHEMA`. State and result text come from `CodexAdapter.detectState`/`extractResult` fed the notify record, so the wire vocabulary (`agent-turn-complete`, `last-assistant-message`) is decoded in exactly one place: `src/adapters/codex.ts`. No `ORCH_AGENT_KEY` ⇒ a non-orch codex session ⇒ exit 0 silently (same rule as the claude shim).

**Synthetic exit code so `done` is reachable.** `detectState` (codex.ts:224–225) resolves a completed event to `done` only when it coincides with a zero exit — `completed ? (exitCode === 0 ? "done" : "idle")` — but codex's notify record carries no exit code, so a completion record alone would resolve to `idle`, never the `done` the presence scenario requires. Because codex fires `agent-turn-complete` only after a settled successful turn, the shim feeds the completion record into `detectState` with a synthetic `exitCode: 0`, producing `done`. The synthesis lives at the shim's call boundary into the adapter, not inside `detectState`, so the parser's contract is untouched (silence is still never read as `done`, `blocked`, or `error`).

*Alternative rejected:* have the shim itself understand codex's JSON. That would smear codex's wire format across `scripts/` and the adapter (law #2 violation). The shim stays a thin presence-writing host; the adapter owns parsing.

### D2 — `installShim()` implemented per adapter, wire format owned by the adapter
- **pi**: build (if missing) and link the bridge bundle into `~/.pi/agent/extensions/` — the logic currently inline in `cmdSetup` (`commands.ts:1680-1694`) plus `buildExtensionBundle`.
- **claude**: install the `SessionStart`/`Stop`/`Notification` hooks — the entire `installClaudeHooks` body (`commands.ts:1553-1609`) relocated onto `ClaudeAdapter.installShim()`, unchanged in behavior (JSON round-trip, runtime probe, stale-entry prune, additive merge).
- **codex**: register the orch shim as codex's notify program. **Primary path:** where the targeted codex version exposes a per-spawn notify mechanism (a spawn flag or environment variable), orch uses that at spawn time and never writes global config — the cleanest law-#5 outcome, since it touches nothing the user owns. **Fallback:** edit the top-level `notify` key in `~/.codex/config.toml` to invoke the built shim under the probed runtime, but ONLY when that key is empty. A pre-existing foreign (non-orch) `notify` value is never overwritten (see D2a). All other codex config is preserved either way.

Each is idempotent (re-running detects its own entry and no-ops) and additive (never touches another provider's config). `installShim()` becomes non-optional in spirit — all three implement it — but stays declared `installShim?()` on the port so the type doesn't force a no-op on some future capability-poor adapter.

**D2a — codex `notify` is a single-program key: refuse, never clobber (law #5).** Unlike claude's hooks array (additive merge) and pi's extensions dir (additive link), codex's `notify` holds exactly one program. Overwriting a user's pre-existing notify program to install orch's would *remove the user's unrelated configuration* — the exact law-#5 failure this change exists to prevent. So the codex fallback path writes `notify` ONLY into an empty key. When a foreign value is present, `installShim()` refuses: it leaves the value untouched, prints a warning naming the conflict, and records a codex presence-capability gap that `orch setup` surfaces and `orch doctor` flags. Codex presence then degrades to headless session-tail parsing with no interactive notify writer — a loud, reported gap, never a silent clobber.

*Alternative rejected for codex:* full TOML parse/serialize round-trip. Orch is deleting its hand-rolled TOML parser (Rule 8), and codex config is the user's file. Instead the codex `installShim` does targeted single-key editing of the one top-level `notify` line (replace when the value already points at orch's shim, insert when the key is absent/empty, refuse when a foreign value is present per D2a), never reformatting the rest of the file; an unparseable/ambiguous file is left untouched with a printed warning, exactly as `installClaudeHooks` bails on bad JSON.

### D3 — A capability-gated session-view port method de-pis `deriveView`
Add `readSessionView(input: SessionViewInput): SessionView | undefined` to the port, implemented only by adapters with `caps.sessionTail`. `SessionView` carries the display fallbacks the view needs: `state?`, `model?`, `provider?`, `thinking?`, `task?`, `lastText?`, `cost?`. `deriveView` replaces its `isPi ? parseSession(...) : null` line with `caps.sessionTail ? adapter.readSessionView({ sessionPath: ent.sessionPath }) : undefined`, and the model/state/cost/task/last fallbacks read from that struct instead of the pi-specific `SessionData`.
- **pi** `readSessionView` wraps the existing `parseSession` (`src/session.ts`) and maps `SessionData` → `SessionView`.
- **claude** `readSessionView` reads the transcript JSONL for `lastText` (its `extractResult` transcript path already exists); state stays presence-driven.
- **codex** `readSessionView` reads the headless `--json` log at the recorded log path (see D3a) and runs it through `detectState`/`extractResult`, wiring the previously-dead parsers into the live view path.

The resolved adapter comes from the spawn registry (`spawned.get(ent.key)?.adapter`) with the presence `agent` field as the fallback — the same source `deriveView`'s `agent` column already uses (`commands.ts:178`).

### D3a — the headless backend records its log path (required, not scanned)
The headless backend already computes a log path and captures the child's stdout/stderr to it (`src/backends/headless/index.ts:189–190`, `$ORCH_DIR/logs/<key>-<timestamp>.log`), but records it **nowhere**: `BackendRegistryRecord` (`src/backends/backend.ts:115`) carries only `backend`/`handle`/`adapter`/`cwd?`, so `readSessionView` has no input and "headless codex result harvested from session output" cannot run. This change adds a `log` field to the headless spawn registry record (and mirrors it into the agent's `status.json`) at spawn time; `deriveView` sources `sessionPath` for a headless agent from that recorded path. The earlier idea of "locate the newest log under the agent's presence log dir" is deleted: it is factually wrong — logs are flat under `$ORCH_DIR/logs/`, never under `$ORCH_DIR/agents/<key>/`, so a per-agent-dir scan finds nothing. When no `log` was recorded, `readSessionView` returns `undefined` and state falls back to `backendStatus`; it never scans and never throws.

*Alternative rejected:* keep `parseSession` in the view and add per-adapter branches. That re-introduces identity branching the change exists to remove.

### D4 — `herdrStatus` → `backendStatus` rename, no compat alias
Rename the field on `Entity` (`entities.ts:17`), its two populate sites (`entities.ts:104,122`), and every `commands.ts` consumer (`:146,236,340,358,1350,1351,1362`) plus the `test/workspace-walls.test.ts` fixture. Golden fixtures `test/golden/status.json` and `status-all.json` update the key. Rule 8: no re-export, no aliased getter — rename and fix callers in one pass.

### D5 — `WORKER_PROMPT_HEADER` becomes adapter-aware at every composition site
The `orch_ask` sentence is appended only for adapters declaring `caps.ask`. Since three copies of the header string exist (`commands.ts:2170`, `work.ts:20`, `daemon/events.ts:10`) and `events.ts` slices it back off by exact length, the header is split into a constant base (`[orch worker] No human watches this pane…`) plus an optional `orch_ask` clause selected by `caps.ask`. `events.ts`'s prefix-strip switches to matching the base prefix so both header variants are recognized. pi/claude/codex all currently declare `ask:false` except pi (`caps.ask` — pi's cap is the one that's true), so only pi workers get the `orch_ask` line; codex/claude workers get the bare header instead of an instruction to call a tool they don't expose.

**Both composition sites, not just the constant.** The prompt is assembled in two places, and both must select the clause from the *resolved target adapter's* caps — editing the constant alone leaves the work loop pi-shaped:
- **Interactive dispatch (`orch run`)** — `workerPrompt` (`commands.ts:2172`) already resolves the target pane; it composes the header from that adapter's `caps.ask`.
- **Work-loop assignment (`orch work`)** — `dispatchTask` (`work.ts:~60`) currently interpolates the bare `WORKER_PROMPT_HEADER` constant and never resolves the assigned agent's adapter. This change has `dispatchTask` resolve the adapter for the assigned agent (spawn registry `adapter` → presence `agent` field, the same source `deriveView`'s agent column uses) and compose the header from its caps. Without this fix a codex or claude worker assigned by `orch work` would still receive pi's `orch_ask` instruction.

### D6 — Claude presence fidelity is documented, not expanded
No new claude hook events. The spec and adapter docstring state plainly that claude presence is coarse: `working` on SessionStart, `blocked` on Notification, `done`/`idle` on Stop, with no mid-run tool/token/cost transitions. Result extraction via the Stop-hook transcript is confirmed working through `ClaudeAdapter.extractResult` and `readSessionView`.

**Claude × headless is an unverified assumption — verify, then spec both outcomes.** Claude presence rides entirely on the `settings.json` hooks (`ClaudeAdapter.hookDriven`), and its headless command is `claude -p …` (`claude.ts:252`). Whether those hooks fire under print mode has never been verified anywhere in the repo — the whole `claude × headless` presence path rests on it. This change adds an explicit verification task (task 6.3): run `claude -p` with hooks installed and observe SessionStart/Stop. Two specced outcomes:
- **Hooks fire** → headless claude presence works exactly as specced (status writer drives state, `result.json` on Stop).
- **Hooks do NOT fire** → result harvest degrades to `ClaudeAdapter.extractResult` over the D3a-recorded `-p` log; `see`/`steer` become *loud documented gaps* — an explicit `no presence in -p mode` status and a non-zero exit on a steer attempt (matching law #3: an unsupported capability fails loudly, never silently no-ops), never a silent `-`.

Either way the behavior is deterministic and reported; the unknown is closed by measurement, not assumption.

## Risks / Trade-offs

- **Codex config TOML editing is fragile, and `notify` is a single-program key** → prefer the per-spawn notify mechanism (no global-config write) where available; the fallback restricts `installShim` to the single top-level `notify` key with line-targeted editing, writing ONLY into an empty key and refusing (warn + report gap) when a foreign value is present (D2a); on any parse ambiguity, leave the file and warn (never reformat or drop user keys). Covered by an idempotency + additivity + foreign-refusal unit test.
- **Headless log path is recorded nowhere today** (`BackendRegistryRecord` has no `log` field; logs are flat under `$ORCH_DIR/logs/`) → this change records the exact log path in the headless spawn registry record and `status.json` at spawn (D3a), and `deriveView` passes it as `sessionPath`; no directory-scan fallback (it would look in the wrong dir). If a record predates the field, session view degrades to `undefined` (state falls back to `backendStatus`), never throws.
- **Claude hooks may not fire under `claude -p`** (never verified) → task 6.3 measures it; if they do not fire, harvest falls back to `extractResult` over the recorded `-p` log and `see`/`steer` fail loudly with an explicit `no presence in -p mode` status + non-zero exit (D6), never a silent `-`.
- **Notify may never fire** (codex killed mid-turn, notify misconfigured) → silence is never read as `blocked`; `detectState` returns `working` for a live process and `error` only on non-zero exit/signal, matching the adapter's existing contract.
- **Relocating `installClaudeHooks` onto the port while `cmdSetup` still calls the old path** → keep the call site working by having it call `claudeAdapter.installShim()`; the body moves, the behavior and output are identical, so no user-visible change and no coupling to the deferred setup rewrite.
- **New port method widens the interface** → `readSessionView` is optional and capability-gated; adapters without `caps.sessionTail` omit it and the view falls back to `backendStatus`, so no adapter is forced into a no-op implementation.

## Migration Plan

Pre-publish project, no live data migration. Sequence: (1) add `readSessionView` + `SessionView` to the port and implement per adapter; (2) add `scripts/codex-notify.ts` + `build:notify` + codex `installShim`; (3) relocate `installClaudeHooks` → `claudeAdapter.installShim()`, repoint the `cmdSetup` call; (4) pi `installShim`; (5) de-pi `deriveView`, rename `backendStatus`, adapter-aware `WORKER_PROMPT_HEADER`; (6) update golden fixtures; (7) run `bun run check`, `check:bridge`, targeted `bun test`, then execute the spec scenarios. Rollback is a git revert — no persisted schema change.
