# Proposal: provider-driven-setup-doctor

## Why

`orch setup` and `orch doctor` are hardcoded to the pi+herdr world. Setup records a **single** chosen harness, then branches on it (`if (harness === "pi") … if (harness === "claude") …`), probes every prerequisite regardless of what the user selected (a claude+tmux user is told `pi MISSING` and offered a pi install), keys its installer table by `bun/pi/claude` with **no codex entry** (selecting codex has no install action), and never invokes the notifier wizard it already ships (dead code, zero `[[notify]]` entries written). Doctor hard-requires pi, always runs the claude-hooks check, and gates the extension check on pi alone — so it lies about the health of any composition that isn't pi, and it selects its provider checks by identity (a law-#3 violation: there is no polymorphic surface to select the right check).

A binding new user requirement compounds this: **setup must be multi-select per axis.** A real machine runs pi **and** claude; the operator wants to install a *set* of adapters and a *set* of backends, then switch which one is active without reinstalling anything. Today setup forces one adapter and one backend, and switching means re-running the whole install.

This violates the settled architecture (design-patterns L4/L4b: setup is the Builder that asks each resolved provider to install its own integration; law #7: doctor derives its checks from the resolved providers and validates declared settings against reality; law #3: callers gate on ports/capabilities, never on adapter id).

## What Changes

- **BREAKING**: Setup becomes the multi-select Builder (L4). The wizard collects a *set* of adapters and a *set* of backends. Setup installs **every** selected provider's integration by calling `resolveAdapter(id).installShim?.()` per selected adapter, then asks which installed provider is the **active default** per axis, and writes both the installed sets and the active defaults to the settings file. The `if (harness === "pi") …`/`if (harness === "claude") …` branches (`commands.ts:1680-1718`) are deleted. Switching the active provider later is a settings edit (or setup re-run) with **no reinstall** — every installed provider's shim is already present.
- Prerequisite probing is keyed to the **selected** providers only, discovered from the provider id (the **id-is-binary invariant**: the adapter/backend id IS its probe binary name), not the fixed `bun/pi/claude` list. `DEP_INSTALLERS` is re-keyed by real provider ids with a declared dependency shape (`needs[]` in explicit order — pi declares `needs:["bun"]`), a **codex entry** added, and a docs-URL form for providers with no automatic installer. `bun` is surfaced only as a provider's declared install dependency, never as an unconditional orch requirement (Rule 6).
- The notifier wizard (`setup/notifiers.ts` — currently dead code) is **wired in** as a setup step. Rather than adding a parallel requirement, this change **modifies** the existing `notifier-adapters` "Notifier setup and doctor validation" requirement to run the step inside the multi-select flow, skipping it on non-interactive / `--yes` runs. (The settings file format/writer are owned by `settings-json-config`; this change consumes its writer.)
- **NEW port surface**: `AgentAdapter` gains a read-side `diagnoseShim?(): CheckResult | Promise<CheckResult>` — the diagnose twin of `installShim()`. The three provider-specific doctor bodies (`checkExtensions`, `checkClaudeHooks`, codex notify-verify) relocate onto the adapters, and doctor calls `resolveAdapter(id).diagnoseShim?.()` **polymorphically for every installed adapter** — removing the law-#3 identity branch.
- **BREAKING**: Doctor becomes composition-aware (law #7). It reads the installed sets + active defaults, derives its checks from **every installed provider** (each self-diagnoses via `diagnoseShim()`), and additionally verifies every pair still live in the fleet via the same `diagnoseShim()`. The hardcoded pi binary requirement (`doctor.ts:99-114`), the unconditional claude-hooks check (`doctor.ts:848`), and the pi-only extension gating (`doctor.ts:527`) are removed. `binaryStatus()` becomes `binaryStatus(ids)`, probing only the selected/installed providers. Doctor **catches** a settings load error and renders it as a failing check rather than dying before checks run. A broken integration for an installed-but-inactive provider still fails doctor.

## Capabilities

### New Capabilities
- `provider-setup`: `orch setup` as the composition-root multi-select Builder — a set selected per axis, prerequisites probed for the selected providers only (id-is-binary), each selected provider installs its own integration via `installShim()`, an active default chosen and recorded per axis, and switching the active default requiring no reinstall.

### Modified Capabilities
- `doctor-config`: `orch doctor` derives its checks from the installed adapters + backends (each provider self-diagnoses via `diagnoseShim()`) and verifies declared settings against reality including every live fleet pair, instead of a hardcoded pi+claude+herdr checklist; it catches a settings load error as a failing check. The `--fix` surface follows the installed providers.
- `notifier-adapters`: the existing "Notifier setup and doctor validation" requirement is modified to run the notifier pick-list inside the multi-select provider-setup flow and skip it on non-interactive / `--yes` runs.

## Impact

- Code: `src/adapters/adapter.ts` (**adds** `diagnoseShim?()` to the `AgentAdapter` port), `src/adapters/pi.ts`/`claude.ts`/`codex.ts` (each gains a `diagnoseShim()` body relocated from doctor), `src/commands.ts` (`cmdSetup` multi-select flow, `DEP_INSTALLERS` re-key + `needs[]` + codex entry, prereq loop, the harness `if` branches, `installClaudeHooks` call site), `src/doctor.ts` (`binaryStatus(ids)`, `checkBins`, relocating `checkClaudeHooks`/`checkExtensions` into `diagnoseShim`, `runDoctor` composition, load-error-as-check guard), `src/setup/notifiers.ts` (wired into the setup flow), `src/setup/doctor-wizard.ts` (unchanged renderer, consumed by provider-derived results).
- New port method `diagnoseShim()` is owned by this change and **mirrors** `installShim()`, whose per-adapter bodies are owned by `adapter-presence-writers`. The two changes coordinate so that for each adapter, diagnose and install target the same integration artifacts (diagnose mirrors install per adapter).
- Consumes ports/writers owned by sibling changes: `installShim()` bodies (`adapter-presence-writers`), the settings file format + installed-sets shape + writer (`settings-json-config`). This change consumes those surfaces; it does not define them.
- Non-goals / deferred: `installShim()` adapter bodies (owned by `adapter-presence-writers`); the settings.json file format, schema, installed-sets shape, and writer (owned by `settings-json-config`); the `commands.ts`/`doctor.ts` monolith file split (owned by `monolith-file-breakdown`). This change edits the setup/doctor logic in place and leaves the physical file reorganization to that change.
